# 🎯 Fast Notice Chrome Extension — Workspace Rules

## 📋 Обзор проекта

Это **Chrome Extension для заметок** с использованием:
- Chrome Storage API, manifest v3, service workers
- Редактор с поддержкой форматирования (заголовки h1-h5, абзацы)
- Автономная работа без внешних зависимостей

---

## 🗂️ Структура проекта

```
fast-notice-chrome-extension/
├── manifest.json          — версия 3.2.1, permissions: storage, tabs
├── background.js          — CRUD заметок, chrome.storage.local
├── editor.js              — редактирование, автосохранение, форматирование
├── texteditor.js          — contenteditable логика, заголовки h1-h5
├── selection.js           — экран выбора заметок
├── editor.html/css        — UI редактора с тултипом для ссылок
├── selection.html/css     — UI выбора заметок
└── tests/
    ├── e2e/               — Playwright E2E тесты
    └── unit/              — Jest/Vitest unit тесты
```

---

## 🔑 Ключевые API (background.js)

| Функция | Описание |
|---------|----------|
| `createNewNote(title, tabId)` | Создаёт новую заметку |
| `getAllNotesSummary()` | Возвращает массив `{id, title, preview, timestamp}` |
| `saveNote(noteId, data)` | Сохраняет заметку в storage |
| `deleteNoteById(noteId)` | Удаляет заметку по ID |

---

## ⚠️ Известные проблемы

- **Баг h1→p**: При смене формата заголовка h1 на параграф p возникает проблема с вложенностью тегов (`<h1><p>...</p></h1>`). Корректный формат: просто `<p>...</p>` без вложенности.
- Тест для этого бага: `tests/e2e/heading-overlap.spec.js`

## 🧪 Тестирование (ОБЯЗАТЕЛЬНО)

### Unit-тесты (Jest/Vitest)
- **Расположение:** `tests/unit/`
- **Покрытие:** 100% для нового кода
- **Запуск:** `npm run test:unit` или `npm test`

### E2E-тесты (Playwright)
- **Расположение:** `tests/e2e/`
- **Запуск:** `npm run test:e2e` или `npx playwright test`

### Визуальные тесты
- **Скриншоты:** `tests/e2e/screenshots/baseline/`
- **Diff:** `tests/e2e/screenshots/diff/`

### Правила тестирования

| Ситуация | Действие |
|----------|----------|
| **Новый функционал** | Создай unit + E2E тесты до реализации (TDD) |
| **Изменение существующего** | Обнови связанные тесты, запусти все |
| **Удаление функционала** | Удали все связанные тесты |
| **Упавший тест** | Исправь код или тест **немедленно** |

## 📝 Правила разработки

### ✅ Обязательно

1. **Тестирование** — каждое изменение должно быть покрыто тестами
2. **Никаких упавших тестов** — не фиксируй код, если тесты падают
3. **Экранирование данных** — не доверяй пользовательскому вводу
4. **Проверка URL** — валидируй внешние ссылки перед использованием в `href`
5. **Автономность** — НЕ используй npm или внешние зависимости без согласования

### 🔄 Цикл разработки

1. **Планирование**  
   - Определи файлы для изменения
   - Определи тесты для создания/обновления
   - Опиши ожидаемое поведение и граничные случаи

2. **Реализация**  
   - Напиши production-код
   - Напиши тесты
   - Используй моки для изоляции API

3. **Верификация**  
   - Запусти unit-тесты
   - Запусти E2E-тесты
   - Убедись, что все тесты зелёные

4. **Финальная проверка**  
   - Линтер и форматтер без ошибок
   - Нет утечек памяти/рендеров
   - Проверка доступности (a11y)

### 🚫 Запрещено

- `xit`, `test.skip`, `// @ts-ignore` без веской причины
- Игнорирование упавших тестов в CI
- Фиксация кода без тестов
- Использование `setTimeout` вместо явных ожиданий (`waitFor`, `findBy*`)
